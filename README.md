# Claude Code プロジェクトパック

受託開発の現場でClaude Codeを使うための、サブエージェント・スキル・運用標準の一式です。IPA共通フレーム2013（SLCP-JCF2013）の工程構造と、V字モデル（要件定義 → 基本設計 → 詳細設計 → 製造 → 単体・結合・総合テスト → 受入 → 納品）に対応しています。そのまま使うものではなく、案件ごとにテーラリングして使う前提で設計しています。

現在のバージョン: **1.0.4**（[CHANGELOG](CHANGELOG.md)）

- サブエージェント23体 / スキル21個 / 標準ドキュメント8点
- 実行可能なツールを同梱（Excel変換・スライド生成・整合性監査・docsレビュー用Webビューア）
- CI 2本（push/PRごとの整合性検査、四半期の棚卸しissue自動起票）

全体像は [docs/flow-map.md](docs/flow-map.md) にMermaidの図4枚（工程×エージェント、ドキュメントの流れ、運用ループ、仮定のライフサイクル）でまとめてあります。最初にこれを見るのが早いです。

## 前提にしている受託開発の事情

このパックは、次の5つを前提に設計しています。

1. ゴールが「動くコード」ではなく「顧客が検収できる成果物一式」であること。工程ごとの成果物、版数、承認記録、トレーサビリティが求められます
2. 品質と信頼を左右するのが、帳票の1円ズレ、夜間バッチのリラン設計、データ移行の切り戻し、レガシーの「現行通り」の解読、障害報告書の第一報といった地味なドメインであること
3. 「モックだけある」「〇〇みたいなものを作ってほしい」「ビジネス目的しか決まっていない」という曖昧なスタートが常態であること。曖昧さを管理する仕組みがないと、AIは勝手な仮定の上に成果物を積み上げます
4. QCDの説明責任。「AIで速くなった気がする」では稟議も検収も通りません
5. 統制がなければAIは事故を量産すること。エンジニアの善意の近道（権限バイパス、秘密情報のプロンプト投入、出所不明のMCP接続）は、注意喚起ではなく仕組みで塞ぐ必要があります

## 全体の見取り図（4層モデル）

AI駆動開発の重心は「プロンプト → コンテキスト → ハーネス → ループ」と移ってきました。このパックは4層すべてを受託開発向けに配置し、下の層を整備しないまま上の層を使うことを統制で禁じています。

| 層 | 重心の時期 | 内容 | このパックでの実装 |
|---|---|---|---|
| プロンプト | 〜2024 | 1回の指示の最適化 | エージェント定義の統一構造（起動時シーケンス・チェックリスト・報告形式） |
| コンテキスト | 2025 | 何を見せるかの情報設計 | CLAUDE.md、docs正本主義、コンテキストヒストリー、仮定マーカー、要約のみの報告 |
| ハーネス | 2026年初頭〜 | 許可設定やhooksで1回の実行を安全にする装備 | permissionsのdeny設定、MCP許可リスト、QCDゲート、ERDの独占管轄、破壊防止の3層防御 |
| ループ | 2026年6月〜 | ハーネス済みのエージェントを自動で回し続ける仕組み | 週次のQCD・診断の定期運転、pm-sync同期、ループ統制（停止手段・上限・人間チェックポイント） |

自組織がどの層まで整備できているかの成熟度診断にも使えます。整備できていない層の一つ上に手を出さないことが、事故防止の原則です。

## 主要な仕組み

エージェントの一覧より先に、何を作り込んであるかを説明します。各エージェントは肩書きではなく、次の仕組みに配線された担保者として定義されています。

### 1. QCD基準 — 「順調です」で報告させない

[qcd-standards.md](docs/standards/qcd-standards.md) に21の基準（品質11・コスト7・納期5）を定義しています。すべての基準に「測るエージェント」と「合否を判定する者」を分けて割り当て、基準IDを引用しない合否宣言は無効というルールにしています。抜粋します。

| ID | 基準 | 既定値 | 測る → 判定 |
|---|---|---|---|
| Q-02 | レビューMust指摘の残存 | 0件 | code-reviewer → leader |
| Q-06 | 不具合密度 | 計画値±50%を外れたら分析。下限割れ（テスト不足の疑い）も異常扱い | test-engineer → quality-performance |
| Q-09 | セキュリティCritical/High | 0件。受容には顧客合意の記録が必須 | security-compliance → 人間 |
| Q-11 | AI成果物の生成と検証の分離 | 100%。自己レビュー禁止 | ai-dev-standardizer → leader |
| C-02 | 工数消化率と進捗率の乖離 | +10%で警告、+20%で人間へ | estimation-specialist → leader |
| D-01 | 進捗の定義 | 成果物完了基準の0か100のみ。「90%完了」は受理しない | documentation-specialist → leader |

週次でレポート（緑・黄・赤）を作り、leaderが判定し、人間が承認します。逆方向の仕組みもあります。3週連続すべて緑なら、基準が形骸化していないかを疑って点検します。守られない標準を量産しないためです。

### 2. 仮定の管理 — 曖昧なまま安全に進む

曖昧なスタートの案件では、未解決の問いを抱えたまま進まざるを得ません。[assumption-management.md](docs/standards/assumption-management.md) の考え方は「エージェントに覚えさせる」のではなく「作業対象から曖昧さを引けるようにする」です。

- 曖昧さはQ-IDで台帳化し、影響範囲（機能ID）と、仮定が外れた場合に壊れる範囲を必ず書く
- 仮定の上に作った箇所には `【仮定: Q-031】` というマーカーを依存箇所ごとに埋め込む
- 実装やテストの担当は、着手前に自分の機能IDで台帳を照合する。仮定が未設定の問いに当たったら作業を止める（勝手に仮定を作らない）
- 解決時は `grep Q-031` で影響箇所を全件列挙してから巻き戻す
- 仮定の上に仮定を積むのは2段まで。3段目は人間にエスカレーションする

### 3. 迂回できない関所

実装エージェントがバイパスできないゲートを4つ置いています。

- ERD・テーブル定義の変更は data-model-specialist だけができる。他は提案のみ
- 新しいライブラリや設計パターンは architecture-guardian の審査を通す。既存で実現できるなら原則却下
- 認証・決済・個人情報に触れたら security-compliance のレビューを受ける。Critical/Highは納品を止める
- 実装と検証は必ず別のエージェントが行う（Q-11）

### 4. トレーサビリティがそのまま監査証跡になる

機能ID → 受入基準 → 設計書の項番 → テスト番号を全成果物でつなぎ、断絶を機械的に検出します。この背骨があるため、「この納品コードは誰が書き、誰がレビューし、どのテストで検証されたか」という監査の問いに、成果物IDから辿って答えられます。

### 5. 出発資産別のプレイブック

案件の始まり方は一つではありません。[/project-init](.claude/skills/project-init/SKILL.md) は契約の形とは別に「スタート地点に何があるか」を判定し、初動と注意点を切り替えます。

| 出発資産 | 初動 | 組み込んだ対策 |
|---|---|---|
| 現行システム | 要件定義より先に機能棚卸し | 「現行通り」を要件と認めない。バグまで踏襲しない |
| 既存コード | 資産評価と前提の外部化 | パックの規約より既存コードの規約を正とする |
| モックだけ | モックから機能一覧と受入基準を逆生成 | モックは見た目の合意にすぎない。業務ルール・異常系・非機能の空白リストを顧客と共有する |
| 類似サービス | 参考サービスの棚卸しと差分定義 | 「あれと同じで」は要件ではない。模倣は法務確認へ |
| RFP・要求仕様 | 曖昧さの4分類検出から要件定義へ | RFPは書いた人の解釈を経ている。出典を遡る |
| ビジネス目的だけ | 要件定義に入らず、業務の構造化と概算から | いきなり画面の話をしない。請負契約ならリスクを明示する |

### 6. 開発規約12種の生成

新規案件には「従うべき既存コード」がありません。放置すると、最初に書いたコードの癖が事実上の規約になります。[/dev-standards](.claude/skills/dev-standards/SKILL.md) は実装開始前に規約の正本12種（コーディングFE/BE、API、DB、ログ、エラー処理、Git運用、テストコード、セキュアコーディング、ディレクトリ、UI、バッチ）を生成します。lintやCIで機械化できる項目は設定に落とし、レビュー観点から外します。書いた規約より、強制される規約です。

### 7. セキュリティと統制

- [ai-security-baseline.md](docs/standards/ai-security-baseline.md) — Claudeのプラン別統制（Enterpriseは組織ポリシーで強制、Teamは共有設定と検知、個人プランは業務利用の可否そのものを人間が判断）。よくある事故パターン10項目とdeny設定のスニペット。そして報告免責の明文化。秘密情報を入れてしまった等の自己申告は責めず、隠蔽だけを重大違反とします
- [enterprise-controls.md](docs/standards/enterprise-controls.md) — 大規模組織向けの上位統制。破壊防止の3層防御（deny設定 → hooks → 2人ルール）、ループ統制（停止手段・上限・無人時間帯の制約）、監査証跡、職務分掌（起案者と承認者を分ける）、協力会社の統制（持ち込みアカウント禁止・同一標準の配布）

### 8. マネジメントとの接続

- [/pm-sync](.claude/skills/pm-sync/SKILL.md) — Jira・Backlog・Redmine・Notion・SlackとのMCP連携。最初に正本ルールを決めます。タスクの状態は外部ツールが正、経緯や理由はdocsが正、両者はIDでつなぐ。議事録の宿題は自動起票し、QCD週次はSlackに配信。即時通知はエスカレーションのみに絞ります
- [feature-team.md](docs/standards/feature-team.md) — 機能ドメインごとにFE/BEのスクワッドを組む並列開発モード。IF定義を先に固定してから両者を並列で走らせます。ERD・レビュー基準・規約を守る横断エージェントは複製禁止です
- [/effort-compression](.claude/skills/effort-compression/SKILL.md) — SES工数の圧縮を計測に変えます。作業棚卸し、AI移管度マップ、ベースライン設定、月次の圧縮率。品質基準を落として得た圧縮は無効として計上しません

### 10. 二つの週次診断と日次の5問

- [/context-health-check](.claude/skills/context-health-check/SKILL.md)（AI側） — 文書間の矛盾、鮮度、外部化漏れ、仮定マーカーと台帳の突合
- [/llm-friendly-check](.claude/skills/llm-friendly-check/SKILL.md)（人間側） — リポジトリの機械診断（巨大ファイル・正本の重複・用語のゆれ）と、指示の質のセルフチェック7問。改善提案は週に1つだけに絞ります
- 日次はセッション終了時の5問だけ。決定はdocsに残ったか。明日の自分は何も覚えていなくても再開できるか。AIに「察してもらった」箇所はないか。未コミットの残骸はないか。マージ済みブランチを離れたか

## 設計思想

1. 多次元分割 — レイヤー（Infra/FE/BE）× 役割（Leader/Coder/Reviewer）× 工程で分ける
2. 横断エージェントの常設 — セキュリティ、アーキテクチャ、データモデル、ドキュメント、品質・性能は工程をまたぐ
3. コンテキスト汚染の防止を最優先 — サブエージェント多用、要約中心の報告、知識の外部化、定期リセット
4. テスト可能性の重視 — 要件は受入基準（AC）中心。曖昧さは4分類で指摘させ、顧客に投げられる質問文まで作らせる
5. 知識の外部化 — 決定・制約・経緯をセッションの外に残し、次のセッションへ引き継ぐ
6. トレーサビリティ — 機能IDで全成果物をつなぎ、断絶を機械検出する

## 導入手順

1. 案件リポジトリにパック一式をコピーします。`.claude/` だけでは動きません。エージェントは `docs/standards/` の基準を参照します
   ```bash
   cp -r .claude/ /path/to/your-project/.claude/
   cp -r docs/ /path/to/your-project/docs/
   cp CLAUDE.md.template .mcp.json.template /path/to/your-project/
   ```
2. `/project-init` を実行します。配置漏れの検査からQCD基準の確定までここで済みます
3. 提案された「最初の一手」から着手します

## 運用の基本

- メインセッションがオーケストレーターです。Claude Codeの仕様上サブエージェントは入れ子にできないため、各エージェントへの委譲と判断の壁打ちはメインセッションが行います
- サブエージェントの報告は要約のみ受け取ります。生ログやコード全文をメインセッションに持ち込みません
- 決定はその日のうちにADRとして外部化します
- 契約、金額、本番適用、顧客への提出、検収判定、リスク受容は人間が行います。各エージェントはこれらに触れる場面で必ずエスカレーションします

## テーラリング — 削るのが正しい

全部入りで使うのは中規模（3〜20人月）だけです。小規模は統合し、大規模は複製と統制強化で対応します。詳細は [tailoring-guide.md](docs/tailoring-guide.md) にありますが、削ってはいけないものが4つあります。

1. コンテキスト汚染防止のルール一式。削った瞬間から品質が劣化し、劣化に気づけなくなります
2. data-model-specialist のERD独占管轄。データの不整合は後工程で最も高くつきます
3. 受入基準（AC）中心の要件定義。テスト可能性は検収可能性に直結します
4. QCD基準そのもの。計測頻度を軽くするのは構いませんが、基準の削除は担保者のいない空席を作ります

削除は `.claude/agents.disabled/` への移動で行い、復活条件を必ず記録します。

## FAQ

**英語プロジェクトで使えますか？**
エージェント定義はすべて日本語です。検収や多段請負といった日本のSI文化が前提のため、翻訳するより、構造だけ参考にして自分たちの文化で書き直すことをおすすめします。

**PPTXでの納品が必須の顧客には？**
スライドはHTML正本からブラウザ印刷でPDF化するのが基本線です。PPTXが必須の場合は、構成と文言までをこのパックで作り、体裁は顧客のスライドマスタ上で仕上げる分担を /slide-deck で定義しています。

**手持ちのエージェント定義と併用できますか？**
できます。ERD独占やレビュー分離などの関所を残したまま、言語特化のエージェントを実装担当の下に追加する構成が相性が良いです。

**1人の小規模案件でも意味がありますか？**
あります。1人の現場こそ「実装した本人がレビューする」事故が起きます。生成と検証の分離（Q-11）と仮定管理の恩恵はむしろ大きく、テーラリングガイドに1人向けの統合構成があります。

**進行中の案件に途中から入れられますか？**
入れられます。/project-init に途中導入の手順があり、現状の外部化（/context-history）から始めます。

**QCDの数値はそのまま使っていいですか？**
使えません。すべて「テーラリングで確定するまでは仮の基準」であり、/project-init が確定を求めます。適正値は業界と案件で変わります。

## コントリビュート

- Issueを歓迎します。特にMCP接続先やプラン別機能など、陳腐化の報告は `maintenance` ラベルでお願いします
- PRは `tools/audit_pack.py` がCIで通ることが条件です（版数の整合もここで検査されます）
- エージェントやスキルを追加する場合は、既存の統一構造と、QCD基準への配線（どの基準の担保者になるか）を揃えてください

バージョニングは [CHANGELOG.md](CHANGELOG.md) 冒頭の方針に従います。要点だけ書くと、エージェント削除・基準IDの変更・正本パスの変更など導入済み案件を壊す変更がMAJOR、追加はMINOR、修正はPATCHです。リリース時はCHANGELOG追記、READMEの版数更新、`git tag vX.Y.Z`、GitHub Releaseの作成までを1セットとします。

## メンテナンス — 動かないときに疑う場所

陳腐化が速い順に、次の3つを疑ってください。

1. `.mcp.json.template` の接続先。MCPサーバーのURLやパッケージ名は変わります
2. `ai-security-baseline.md` のプラン別機能。Claudeの管理機能は更新されます
3. 4層モデルのハーネス・ループ層の記述。Claude Code本体の進化と乖離しえます

放置されないよう、自動化を同梱しています。push/PRごとに整合性監査が走り、四半期（1・4・7・10月の1日）に棚卸しissueが自動で立ちます。issueにはMCP接続先の死活確認の結果が添付されます。なお、GitHubは60日間コミットがないとスケジュール実行を止めるため、四半期issueが来ないこと自体が点検のサインです。

## おわりに — このパックが自動化しないもの

ここまでエージェントと基準の話をしてきましたが、プロジェクトを成功させるのは、最後まで人と組織に向き合うことだと考えています。

顧客の「大丈夫です」の声色が先週と違う。決裁者が資料のどのページで目を止めたか。ベテランの運用担当者が、新画面の説明のあいだ一度も頷かなかった。メンバーの「ちょっと詰まっていて」が3日続いている。プロジェクトを壊すものも救うものも、たいていはこうした構造化できないインサイトの中にあります。それを拾い続ける仕事に、テンプレートも基準IDもありません。

人と向き合う仕事は、技術や仕組みに逃げ込めないという意味で、いちばん逃げ場がありません。だからこそ本質だと思います。このパックが定型を引き受けるのは、その仕事のための時間をつくるためです。QCDレポートが緑でも、顧客の表情が曇っていたら、信じるべきは表情のほうです。
