# Claude Code Project Pack for 日本のSI事業者

日本のSIer / 受託開発の**工程・検収・統制**をまるごとClaude Codeに載せるためのプロジェクトパックです。
IPA 共通フレーム2013（SLCP-JCF2013）の工程構造と、日本のSI現場で一般的なV字モデル
（要件定義 → 基本設計 → 詳細設計 → 製造 → 単体テスト → 結合テスト → 総合テスト → 受入 → 納品）に対応し、**案件ごとにテーラリングして使う**ことを前提にしています。

**構成: サブエージェント23体 / スキル21個 / 標準ドキュメント8点 / 実行可能ツール（Excel変換・スライド生成・整合性監査） / CI 2本 / CLAUDE.md・MCP接続テンプレート**

全体像は [docs/flow-map.md](docs/flow-map.md)（Mermaid 4枚: 工程×エージェント / ドキュメント正本フロー / 運用ループ / 仮定ライフサイクル）から掴むのが最短です。

## なぜ汎用エージェント集では足りないのか

[VoltAgent](https://github.com/VoltAgent/awesome-claude-code-subagents) のような優れた汎用エージェント集は存在します。それでも本パックを作ったのは、日本のSI受託には汎用集でカバーされない構造問題が5つあるからです。

1. **検収がゴール** — 「動くコード」ではなく「顧客が検収できる成果物一式」が納品物。工程ごとの成果物・版数・承認記録・トレーサビリティが要る
2. **日本SI固有ドメイン** — 帳票（1円ズレ・現行踏襲）、夜間バッチ（リラン・突き抜け）、データ移行（切り戻し）、レガシー解析（「現行通り」の解読）、障害報告書（第一報の作法）は汎用集にない
3. **曖昧なスタートが常態** — 「モックだけある」「〇〇みたいなの作って」「ビジネス目的だけ」から始まる。曖昧さを構造化して持ち運ぶ仕組みがないと、AIは勝手に仮定して壊れる
4. **QCDの説明責任** — 「AIで速くなった気がする」では稟議も検収も通らない。基準値・実測・判定者が要る
5. **統制がないとAIは事故を量産する** — エンジニアの善意の近道（権限バイパス・秘密情報のプロンプト投入・野良MCP）を仕組みで塞ぐ必要がある

## 4層モデル — このパックの見取り図

AI駆動開発の重心は「プロンプト → コンテキスト → ハーネス → ループ」と進化してきました。本パックは4層すべてを受託開発向けに配置し、**下の層を飛ばして上の層を使うことを統制で禁じています**（ハーネスなしのループ運転は禁止、など）。

| 層 | 重心の時期 | 何のことか | 本パックでの実装 |
|---|---|---|---|
| プロンプト | 〜2024 | 1回の指示・聞き方の最適化 | 各エージェント定義（起動時シーケンス・チェックリスト・報告形式の統一構造） |
| コンテキスト | 2025 | 何を見せるかの情報設計 | CLAUDE.md / docs正本主義 / コンテキストヒストリー / 仮定マーカー / 要約のみ報告 |
| ハーネス | 2026初頭〜 | 許可設定・hooks・ガードレールで1回の実行を安全に動かす装備 | permissions deny / MCP許可リスト / QCDゲート / ERD独占管轄 / 破壊防止3層防御 |
| ループ | 2026年6月〜 | ハーネス済みエージェントを自動で回し続ける仕組み | 週次QCD・診断の定期運転 / pm-sync同期 / ループ統制（kill switch・上限・人間チェックポイント） |

成熟度診断にも使います: 自組織がどの層まで整備できているかを ai-dev-standardizer が点検し、**整備できていない層の一つ上に手を出さない**のが事故防止の原則です。

## 特徴的な仕組み（このパックの本体）

一覧表の前に、何を作り込んであるかを説明します。エージェントは「肩書き」ではなく、以下の仕組みに配線された担保者です。

### 1. QCD基準 — 「順調です」を禁止する

[docs/standards/qcd-standards.md](docs/standards/qcd-standards.md) に**21基準**（Q11 / C7 / D5）を定義。全基準に「担保エージェント（測る人）」と「判定者（合否を言う人）」を分離して割り当て、**基準IDを引用しない合否宣言を無効**としています。抜粋:

| ID | 基準 | デフォルト値 | 担保 → 判定 |
|---|---|---|---|
| Q-02 | レビューMust指摘の残存 | 0件 | code-reviewer → leader |
| Q-06 | 不具合密度 | 計画値±50%外で分析発動（**下限割れ=テスト不足も異常**） | test-engineer → quality-performance |
| Q-09 | セキュリティCritical/High | 0件（受容は顧客合意記録必須） | security-compliance → **人間** |
| Q-11 | AI成果物の生成/検証分離 | 100%（自己レビュー禁止） | ai-dev-standardizer → leader |
| C-02 | 工数消化率−進捗率の乖離 | +10%警告 / +20%人間エスカレーション | estimation-specialist → leader |
| D-01 | 進捗の定義 | 成果物完了基準の0/100のみ（**「90%完了」を受理禁止**） | documentation-specialist → leader |
| D-03 | 完了日の再予測 | 実績ペース由来のみ（希望日ベース禁止） | leader → 人間 |

週次でQCDレポート（緑/黄/赤）→ leader判定 → 人間承認。**3週連続オール緑なら基準の形骸化を疑って点検する**逆方向の仕組みも入っています（守られない標準を量産しないため）。

### 2. 仮定管理 — 曖昧なまま安全に進むための構造

スタートが曖昧な案件では未解決の曖昧さを抱えて進まざるを得ません。[docs/standards/assumption-management.md](docs/standards/assumption-management.md) は「エージェントに覚えさせる」のではなく「**作業対象から曖昧さを引ける**」構造にしています:

- 曖昧さは Q-ID＋**影響範囲（機能ID必須）**＋**爆風半径**（外れたら何が壊れるか）で台帳化
- 仮定の上に作った箇所には `【仮定: Q-031】`（コードなら `// 仮定: Q-031`）を**依存箇所ごとに**埋め込む
- coder・テスト系は**着手前に担当機能IDで台帳をフィルタ**。仮定未設定のQに命中したら停止（勝手に仮定を作らない）
- 解決時は `grep Q-031` で巻き戻し対象を**全件機械列挙** → 修正タスク化してからクローズ
- 仮定の多段積みは**2段まで**。3段目は人間エスカレーション（巻き戻し不能の塔を建てない）

### 3. 強制ゲート — Coderが迂回できない4つの関所

- **ERD・テーブル定義は data-model-specialist の独占管轄**（他エージェントは変更提案のみ。全機能への影響調査が省略不可）
- **新規ライブラリ・パターンは architecture-guardian の審査**（既存で実現できるなら却下・Team Familiarity重視・例外は期限付き）
- **認証・決済・個人情報を触ったら security-compliance のレビュー**（Critical/Highは納品ブロック権限つき）
- **実装と検証は別エージェント**（Q-11。実装したAIに自己レビューさせない）

### 4. トレーサビリティ背骨 = そのまま監査証跡

`機能ID → AC → 設計書項番 → テストNo` を全成果物で連結し、断絶を機械検出します。この背骨があるため、「この納品コードは誰が書き、誰がレビューし、どのテストで検証されたか」に**成果物IDから辿って回答**でき、大規模SIerの監査対応にそのまま使えます。

### 5. 出発資産プレイブック — 「始まり方」は6通りある

[/project-init](.claude/skills/project-init/SKILL.md) は案件種別（契約の形）と**出発資産（スタート地点に何があるか）**を別軸で判定し、パターン別に初動と罠を切り替えます:

| 出発資産 | 初動（要件定義の前にやること） | 組み込まれた罠対策 |
|---|---|---|
| 現行システム | legacy-modernizerで機能棚卸し（廃止候補の特定が先） | 「現行通り」を要件と認めない。バグまで踏襲しない |
| 既存コード（エンハンス） | 資産評価→既存前提の外部化→影響分析 | パックの規約より**既存コードの規約を正**とする |
| モックだけ | モックから機能一覧とACを逆生成 | モックは「見た目の合意」— 業務ルール・異常系・非機能の**空白リスト**を顧客と共有 |
| 類似サービス | 参考サービスの機能棚卸し→Must/Want→**自業務との差分定義** | 「あれと同じで」は要件ではない。模倣は要法務確認 |
| RFP・要求仕様 | 曖昧さ4分類検出→要件定義 | RFPは書いた人の解釈を経ている。出典を遡る |
| ビジネス目的だけ | **要件定義に入らない**。業務構造化→線引き→概算±50%→投資判断材料 | いきなり画面の話をしない。請負契約ならリスクフラグ |

### 6. 開発規約12種 — 「1人目のcoderの癖」を規約にしない

新規案件には「従うべき既存コード」がありません。[/dev-standards](.claude/skills/dev-standards/SKILL.md) が coder稼働前に規約正本12種（FE/BE・API・DB・ログ・エラー処理・Git・テストコード・セキュアコーディング・ディレクトリ・UI・バッチ）を生成します。原則は「**書いた規約より強制される規約**」— lint/CIで機械化できる項目は設定に落とし、レビュー観点から外す。同じ指摘が3回出たら「人ではなく規約の欠陥」として改訂します。

### 7. セキュリティ・統制 — 善意の近道を仕組みで塞ぐ

- [ai-security-baseline.md](docs/standards/ai-security-baseline.md): **Claudeプラン別統制**（Enterprise=組織ポリシー強制 / Team=共有設定＋検知 / 個人=業務利用可否そのものを人間承認）＋**穴あけパターン10項目**（権限バイパス・secrets読ませ・野良MCP・本番データ投入…）と deny設定スニペット。**報告免責**（自己申告は責めない・隠蔽のみ重大違反）を明文化
- [enterprise-controls.md](docs/standards/enterprise-controls.md): 大規模SIer向け上位統制 — **破壊防止3層防御**（deny → hooks → 2人ルール）、**ループ統制**（kill switch・上限・無人時間帯制約）、監査証跡、職務分掌（起案者≠承認者）、**協力会社統制**（持ち込みアカウント禁止・同一標準配布）、教育・段階的権限

### 8. マネジメント連携 — 人間が普段のツールで見える

- [/pm-sync](.claude/skills/pm-sync/SKILL.md): Jira / Backlog / Redmine / Notion / Slack とのMCP連携。**正本ルールを最初に決める**（タスクの状態=外部ツール、「なぜ」=docs、IDで相互リンク）。議事録の宿題は自動起票、QCD週次はSlack配信、即時通知はエスカレーションのみ（通知疲れ対策）
- [feature-team.md](docs/standards/feature-team.md): 機能ドメインごとにFE/BEスクワッドを組む並列開発モード。**IF契約ファースト**（api-designerがIFを固定→FE/BEがモックで並列実装）と**横断エージェント複製禁止則**（ERD・レビュー基準・規約の方言化を防ぐ）
- [/effort-compression](.claude/skills/effort-compression/SKILL.md): SES工数圧縮の計測化。作業棚卸し→AI移管度マップ→ベースライン→月次圧縮率。**Q基準悪化を伴う圧縮は無効計上**（テストを薄くして「圧縮」は粉飾）

### 10. 三つの診断 — AIと人間の両方を定期点検

- [/context-health-check](.claude/skills/context-health-check/SKILL.md)（週次・AI側）: 文書間矛盾・鮮度・外部化漏れ・仮定マーカーと台帳の突合
- [/llm-friendly-check](.claude/skills/llm-friendly-check/SKILL.md)（週次・**人間側**）: リポジトリの機械診断（巨大ファイル・正本重複・用語ゆれ）＋指示の質セルフチェック7問＋運用習慣。**改善提案は週1つだけ**（全部直せという診断は誰も直さない）
- 日次3問（セッション終了時30秒）: 決定はdocsに残ったか / 明日の自分は何も覚えてなくても再開できるか / AIに「察してもらった」箇所はないか

## 設計思想

1. **多次元分割** — レイヤー（Infra/FE/BE）× 役割（Leader/Coder/Reviewer）× 工程（要件定義〜納品）でサブエージェントを分割
2. **横断サブエージェント** — セキュリティ、アーキテクチャ、データモデル（ERD）、ドキュメント、品質・性能は工程をまたぐため専用サブを常設
3. **コンテキスト汚染防止を最優先** — サブエージェント多用・要約中心の報告・知識の外部ドキュメント化・定期リセット
4. **テスト可能性重視** — 要件は Acceptance Criteria 中心。曖昧さはAIに4分類（未定義/多義/暗黙/矛盾）で指摘させ、顧客に投げられる質問文まで作らせる
5. **知識の外部化** — Documentation Specialist と「コンテキストヒストリー」でセッションを超えて決定・制約を引き継ぐ
6. **トレーサビリティ** — 機能ID → AC → 設計書項番 → テストNo を全成果物で連結し、断絶を機械検出する

## サブエージェント一覧（.claude/agents/ 23体）

各エージェントは「起動時シーケンス → ドメイン別チェックリスト → ワークフローフェーズ → 連携マップ → 報告形式（要約のみ） → 品質基準」の統一構造です。

### 統括・要件
| エージェント | 役割 |
|---|---|
| leader | 統括PL。工程完了判定（QCD基準ID引用必須）・横断調整・トレードオフ裁定・エスカレーション判断 |
| requirements-analyst | AC作成・曖昧さ4分類検出・非機能要件（IPA非機能要求グレード6大項目） |

| estimation-specialist | WBS積上げ＋類推のクロスチェック・前提条件/除外事項・リスクバッファ・実行中のコスト統制（C基準） |

## 導入方法

1. 案件リポジトリにパック一式をコピーする（**`.claude/` だけでは動かない** — エージェントは `docs/standards/` の基準を参照する）
   ```bash
   cp -r .claude/ /path/to/your-project/.claude/
   cp -r docs/ /path/to/your-project/docs/
   cp CLAUDE.md.template .mcp.json.template /path/to/your-project/
   ```
2. **`/project-init` を実行する**（配置漏れ検査→ヒアリング→テーラリング→CLAUDE.md生成→QCD確定→セキュリティ統制適用。手動なら `docs/tailoring-guide.md`）
3. `/project-init` が提案する「最初の一手」から着手する（出発資産プレイブック参照）

## 運用の要点（詳細は CLAUDE.md.template）

- **メインセッション = オーケストレーター**。Claude Code の制約上サブエージェントはサブエージェントを呼べないため、「Leaderが専門サブを階層的に呼ぶ」構造は *メインセッションが各サブへ委譲し、判断はleaderに壁打ちする* 形で実現する
- **サブエージェントの報告は要約のみ**（全エージェントの「報告形式」節で強制。生ログ・コード全文をメインに持ち込まない）
- **決定はその日のうちにADR化**、週次で2診断、月1回の大リセット＋ヒストリー引き継ぎ
- **人間にしかできないこと**は各エージェントが明示的にエスカレーションする: 契約・金額・本番適用・顧客提出・検収判定・リスク受容

## テーラリング — 削るのが正しい

全部入りで使うのは中規模（3〜20人月）だけです。小規模は統合し、大規模は複製・統制強化します（[tailoring-guide.md](docs/tailoring-guide.md)）。ただし**削減禁止**が4つ:

1. コンテキスト汚染防止ルール一式（削った瞬間から品質が劣化し、劣化に気づけなくなる）
2. data-model-specialist のERD独占管轄（データ不整合は後工程で最も高くつく）
3. AC中心の要件定義（テスト可能性＝検収可能性）
4. QCD基準そのもの（計測頻度の軽量化は可。基準の削除は不可 — 担保者が空席の基準を作らない）

削除は `.claude/agents.disabled/` への移動＋**復活条件の記録**で行います（例:「帳票要件が出たら report-specialist を戻す」）。

## FAQ

**Q. 英語プロジェクトで使える？** — エージェント定義はすべて日本語です。工程文化（検収・帳票・多段請負）が日本SI前提なので、翻訳より「構造だけ借りて自分の文化で書き直す」ことを推奨します。

**Q. PPTX納品が必須の顧客は？** — スライドはHTML正本→ブラウザ印刷PDFが基本線。PPTX必須なら下書き（構成＋文言）まで本パック、体裁は顧客マスタ上で仕上げる分担を /slide-deck が定義しています。

**Q. VoltAgent等の汎用集と併用できる？** — できます。本パックの横断ゲート（ERD独占・レビュー分離・QCD）を残したまま、言語特化エージェント（rust-engineer等）をcoder系の下請けとして追加するのが相性の良い構成です。

**Q. 小規模・1人でも意味ある？** — あります。1人こそ「実装した本人がレビューする」事故が起きるので、Q-11（生成/検証分離）と仮定管理の恩恵が大きい。tailoring-guideの小規模構成（エージェント6体まで統合）を使ってください。

**Q. 既存案件の途中から入れられる？** — 入れられます。/project-init に「既存案件への途中導入」手順があり、現状資産の外部化（/context-history）から始めます。

**Q. QCDの数値はこのまま使っていい？** — ダメです。全基準値は「テーラリングして確定するまで仮基準」であり、/project-init が確定を強制します。業界・案件で適正値は変わります。

## コントリビュート

## メンテナンス — 動かない場合はここを疑え

陳腐化が速い順に:

1. **`.mcp.json.template` の接続先** — MCPサーバーのURL・パッケージ名は変わる。mcp registry で最新を確認して差し替える
2. **`docs/standards/ai-security-baseline.md` のプラン別機能** — Claudeの管理機能（managed settings等）は更新される。最新のAnthropicドキュメントを正とする
3. **README 4層モデルのハーネス/ループ層** — Claude Code本体の機能進化と乖離しうる

放置されないように、リポジトリ側に自動化を同梱しています:

| 仕組み | 内容 |
|---|---|
| `.github/workflows/consistency.yml` | push/PRごとに `tools/audit_pack.py` を実行。社名・個人名の混入検査は Secrets `AUDIT_KEYWORDS` で追加 |
| `.github/workflows/quarterly-review.yml` | 四半期cron（1/4/7/10月1日）で棚卸しissueを自動起票。MCPエンドポイント死活の機械確認つき |

※ cron はGitHubのデフォルトブランチでのみ動作。フォーク運用時は Actions を有効化すること。60日コミットがないとGitHubがscheduleを止めるため、「四半期issueが来ないこと」自体が点検サインです。

## 最後に — このパックが自動化しないもの

ここまで23体のエージェントと基準の話をしてきたが、プロジェクトを成功させるのは最後まで**人と組織に向き合うこと**である。

顧客の「大丈夫です」の声色が先週と違うこと。決裁者が資料のどのページで目を止めたか。ベテラン運用者が新画面の説明中に一度も頷かなかったこと。メンバーの「ちょっと詰まってて」が3日続いていること。——プロジェクトを壊すものも救うものも、たいていこういう**構造化できない不確実なインサイト**の中にいる。それを拾い続けることに、テンプレートも基準IDもない。

人と向き合うのは、技術や仕組みに逃げ込めないという意味で最も逃げ場のない仕事であり、だからこそ本質である。このパックが定型を引き受けるのは、あなたがその仕事から逃げないための時間を作るためだ。QCDレポートが緑でも、顧客の表情が曇っていたら、信じるべきは表情のほうである。
