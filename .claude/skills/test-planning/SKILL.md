# テスト計画スキル

テスト文書一式を `docs/test/` 配下に作成する。作業は test-engineer サブエージェントへ委譲する。

## V字対応（必須）

| レベル | 検証根拠 | 成果物 |
|---|---|---|
| 単体 | 詳細設計書 | `docs/test/unit/<機能ID>.md` |
| 結合 | 基本設計書 | `docs/test/integration/<シナリオID>.md` |
| 総合 | 要件定義書(AC) | `docs/test/system/<シナリオID>.md` |
| 受入支援 | 要件定義書(AC) | `docs/test/uat-support/` |

## 手順

1. 対象工程・対象機能を確認し、テスト計画書を先に作る（初回のみ）
2. test-engineer にテスト仕様書を作成させる（根拠文書の項番トレース必須）
3. トレーサビリティ検査: 「根拠なきテスト」「テストなき要件/設計項目」の両方向を検出し、リストを報告させる
4. 実施後、成績書を記録する。不合格は `docs/issues.md` に起票する

## テスト計画書テンプレート（`docs/test/test-plan.md`）

- 対象範囲・対象外範囲（明記必須）
- テストレベルごとの: 開始条件 / 完了条件（消化率・不具合密度・未解決Critical件数） / 環境 / データ方針（個人情報を含む本番データ使用禁止） / 体制（AI実施と人間実施の分担）
- 不具合管理ルール（起票先・重要度定義・再テスト手順）

## テスト仕様書 兼 成績書テンプレート

| No | 根拠項番 | 観点分類 | 前提条件 | 手順 | 期待結果 | 合否 | 実施日 | 実施者 | エビデンス |
|---|---|---|---|---|---|---|---|---|---|

- 観点分類: 正常系 / 境界値 / 異常系 / 権限 / 同時実行 / データバリエーション
- 期待結果は観測可能な事実のみ（「正しく〜される」禁止）
- エビデンス（スクリーンショット・ログ）は `docs/test/evidence/` に保存しパスを記録

## 完了条件（工程共通）

- トレーサビリティ両方向に漏れゼロ、または漏れの理由が記録済み
- テスト計画書の完了条件を満たす、または未達理由と対応方針をleader判断で記録済み
- 成績書に実施日・実施者・エビデンスが全件記録されている
