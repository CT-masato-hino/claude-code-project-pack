# テスト計画スキル

テスト文書一式を `docs/03-test/` 配下に作成する。作業は test-engineer サブエージェントへ委譲する。

## V字対応（必須）とドラフト作成時期（W字・シフトレフト）

| レベル | 検証根拠 | ドラフト作成時期 | 成果物 |
|---|---|---|---|
| 単体 | 詳細設計書 | 詳細設計の完了前（単体テスト観点として詳細設計書に含める） | `docs/03-test/unit/<機能ID>.md` |
| 結合 | 基本設計書 | 基本設計の完了前 | `docs/03-test/integration/<シナリオID>.md` |
| 総合 | 要件定義書(AC) | 要件定義の完了前（テスト計画書全体もこのタイミング） | `docs/03-test/system/<シナリオID>.md` |
| プレUAT（任意） | —（合否記録なし） | 総合テスト完了後・受入テスト前 | `docs/03-test/pre-uat/`（ガイドとメモ） |
| 受入支援 | 要件定義書(AC) | 総合テスト開始前 | `docs/03-test/uat-support/` |

- **テスト設計を実装後に回さない。** ドラフトの目的は、テスト観点から左側工程へフィードバックを返すこと（ACのテスト容易性・完了条件の実測可能性・テストデータの準備可否）。「テスト可能なACを書く」規律の受け手を工程の最後まで空席にしない
- ドラフトは対応する定義・設計工程の完了条件に含まれる（leaderのゲート判定・deliverables-catalog と連動）
- 各テスト工程の開始時に、ドラフトを実施可能な確定版へ更新する（環境・データ・実施者を確定）

## 手順

1. 対象工程・対象機能を確認し、テスト計画書を先に作る（初回のみ。**要件定義の完了前に着手する**）
2. test-engineer にテスト仕様書（左側工程ではドラフト）を作成させる（根拠文書の項番トレース必須）
3. ドラフト作成で検出したテスト困難点（観測不能なThen・実測不能な完了条件・準備不能なデータ）を、対応する定義・設計工程に**完了前に**フィードバックする
4. トレーサビリティ検査: 「根拠なきテスト」「テストなき要件/設計項目」の両方向を検出し、リストを報告させる
5. 実施後、成績書を記録する（**冒頭の数値サマリー表を必ず埋める**）。不合格は `docs/issues.md` に起票する

## テスト計画書テンプレート（`docs/03-test/test-plan.md`）

- 冒頭に「レビュアー向けサマリ」欄（deliverables-catalog の共通規約）
- 対象範囲・対象外範囲（明記必須）
- テストレベルごとの: 開始条件 / 完了条件（**消化率・不具合密度・未解決Critical件数を数値で**） / 環境 / データ方針（個人情報を含む本番データ使用禁止） / 体制（AI実施と人間実施の分担）
- 不具合管理ルール（起票先・重要度定義・再テスト手順）

## テスト仕様書 兼 成績書テンプレート

**冒頭に数値サマリー表を必須欄として置く**（合否の○×だけで完成扱いにしない。計画書の完了条件と突合できる形にする）:

```markdown
## 数値サマリー（必須）
| 項目 | 計画値 | 実測値 | 差分 |
|---|---|---|---|
| 実施件数（実施/総数） | | | |
| 合格 / 不合格 / スキップ | | | |
| 消化率 | | | |
| 実行時間 | | | |
| カバレッジ（計測している場合） | | | |
| 不具合密度（Q-06） | | | |
```

- この表の値が週次QCDレポート・ゲート判定記録（project-phase.md「根拠」列）への転記元になる（**数値なしの合否宣言は無効** — qcd-standards 運用原則）

| No | 根拠項番 | 観点分類 | 前提条件 | 手順 | 期待結果 | 合否 | 実施日 | 実施者 | エビデンス |
|---|---|---|---|---|---|---|---|---|---|

- 観点分類: 正常系 / 境界値 / 異常系 / 権限 / 同時実行 / データバリエーション
- 期待結果は観測可能な事実のみ（「正しく〜される」禁止）
- エビデンス（スクリーンショット・ログ）は `docs/03-test/evidence/` に保存しパスを記録
- **エビデンス保存の規約**: ログ・スタックトレースを保存する前に、ローカル絶対パス（OSユーザー名を含むパス）は `<repo>` に置換する（実名・ローカルパスの成果物混入防止。audit_pack.py の混入検査対象）

## ローカルUI目視チェック（UIあり案件 — 製造・単体完了ゲートの条件）

E2E・typecheck が通っても「見た目が崩れている」は検出できない。UIあり案件では製造・単体完了ゲートの条件として実施する（バッチ/API専業は適用しない）:

1. frontend-coder が変更画面のスクリーンショット一式を `docs/03-test/ui-check/` に保存する（撮るのはAI）
2. 人間がUI品質チェックリストに○×をつける（見るのは人間 — Q-11 生成/検証分離と同型）:
   - [ ] HTMLモック（基本設計承認時）との乖離がないか
   - [ ] 空（0件）・エラー・ローディングの3状態が実データで確認できるか
   - [ ] 主要ブレークポイントでレイアウトが破綻しないか
3. 記録（UI確認記録）を leader のゲート判定で突合する（deliverables-catalog 製造・単体の◎成果物）

## プレUAT（任意工程 — 総合テスト完了後・受入テスト前）

**実環境構築という重い投資の前に、ユーザー視点の一気通貫操作でUX不満と取りこぼし不具合を安く回収する**工程。「ドッグフーディング」ではなくSI標準語彙に合わせ「プレUAT」と呼ぶ。

- **位置づけ**: 任意実施・**合否記録なし・検収対象外**。気づきは課題起票候補としてメモし（`docs/03-test/pre-uat/notes-YYYYMMDD.md`）、leader のトリアージ（記録と起票の分離）で扱う
- **受入テストとの区別**: 受入テスト（UAT）は顧客が実環境で実施し検収に直結する。プレUATは開発側がローカルで実施し、記録は課題候補のみ。この区別を `docs/03-test/pre-uat/README.md`（プレUATガイド）に明文化してから始める
- **コンプラ停止点はローカルでも有効**: 実在サイト・実在情報へのアクセス禁止、本番データ使用禁止などの停止基準は、ローカル実行を理由に緩めない

### 前提タスク（プレUATを成立させる環境整備）

1. **ローカル起動経路**: 全画面をローカルで開ける状態にする。本番認証が邪魔をする場合のローカル認証迂回は、ai-security-baseline「安全な穴あけ設計（ローカル認証迂回）」のパターン必須（ローカル専用設定ファイル限定・本番混入のトリップワイヤー・監査痕跡・security-compliance レビュー）
2. **モック連携先の単体起動**: テスト内起動前提のダミーサーバー等は、手動確認用に単体起動できるようスクリプト化する
3. **シードデータ**: DBが空だと画面確認が成立しない。フロー操作では作れない状態（未実装機能由来の空・特定パターン依存の失敗系）も含め、**全画面・全状態を網羅するシード＋リセット手順**を整備する（実在情報不使用）
4. **実施可否表**: 機能×「ローカルで確認できる/できない（理由）」の表をガイドに含める
5. 環境分離の確認: ローカル専用設定がテスト実行環境・本番ビルドに漏れないことをデプロイ前チェックで検証する

## 完了条件（工程共通）

- トレーサビリティ両方向に漏れゼロ、または漏れの理由が記録済み
- テスト計画書の完了条件を満たす、または未達理由と対応方針をleader判断で記録済み
- 成績書に**数値サマリー表**と実施日・実施者・エビデンスが全件記録されている
- UIあり案件（製造・単体）: UI確認記録に人間の○×が記録されている
