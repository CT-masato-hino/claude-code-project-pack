# プロジェクト初期化スキル（スターター）

パックを「そのまま全部入り」で使い始めるのを防ぎ、案件に合った最小構成に**根拠つきで**チューニングする。
このスキルはメインセッションが対話で進める（サブエージェントはヒアリングの往復ができない）。記録作成は ai-dev-standardizer に委譲する。

設計の要: **「案件種別（契約の形）」と「出発資産（スタート地点に何があるか）」は別の軸**。
初動工程・ヒアリングの深さ・特有の罠は出発資産で決まる。

## フェーズ0: 配置検査（ヒアリングの前に機械的に確認）

- `docs/standards/`（qcd-standards / assumption-management / ai-security-baseline / feature-team）と
  `docs/deliverables-catalog.md` `docs/tailoring-guide.md` `docs/ipa-process-mapping.md` が案件リポジトリに存在するか確認する
- **欠けていればパック原本からコピーしてから進む**（エージェントはこれらを照合先として参照するため、欠けたまま進むと全基準参照が切れる）
- `CLAUDE.md.template` / `.mcp.json.template` の有無も確認する

## フェーズ1: ヒアリング（ブロックごとに確認。一度に全部聞かない）

### ブロックA: 案件の骨格
1. 案件種別: 新規開発 / リプレース / エンハンス（既存への機能追加・改修） / 保守運用
2. 契約形態: 請負 / 準委任 / ラボ（工程ごとに異なる場合は内訳。**上流が曖昧な案件で請負ならリスクフラグ**）
3. 受託範囲: どの工程からどの工程まで
4. 規模感: 概算人月・想定画面数・チーム人数

### ブロックB: 出発資産の判定（このスキルの心臓部。複数選択可）
5. スタート地点に何があるか:
   - [ ] 現行システム（稼働中。リプレース/参照対象）
   - [ ] 既存コードベース（エンハンス対象。自社開発か他社開発か）
   - [ ] モック・プロトタイプ・デザインカンプ
   - [ ] 類似システム・参考サービス（「〇〇みたいなものを」）
   - [ ] RFP・要求仕様書・要件リスト
   - [ ] ビジネス目的・課題感のみ（文書なし）
6. 各資産の状態: ドキュメントの有無と鮮度 / コードは読めるか / モックはどこまで作り込まれているか / 参考サービスは触れるか

### ブロックC: 技術・機能特性
7. 技術スタック（決定済みか、これから選定か）
8. 帳票 / 9. バッチ・締め処理 / 10. 外部連携 / 11. データ移行 の有無

### ブロックD: 顧客・体制
12. 顧客標準（成果物様式・Excel納品指定）と技術リテラシー・意思決定速度
13. 業種規制: 金融(FISC) / 医療(3省2ガイドライン) / 官公庁 / なし
14. 運用の引き取り手 / 15. AI利用の顧客方針（開示要否・データ投入制限。不明なら起票）
16. タスク管理・連絡ツール: Jira / Backlog / Redmine / Notion / Slack の利用有無と、AI同期の可否（→ 使う場合は /pm-sync でセットアップ。外部SaaSへの案件情報送信可否は15と併せて確認）
17. **Claudeのプラン: Enterprise / Team / 個人（Pro/Max）**（→ docs/standards/ai-security-baseline.md のプラン別統制を適用する。**個人プランの場合は業務利用可否の人間承認が下りるまで顧客データを扱う作業を開始しない**。機密度の高い業種（Q13が金融/医療/官公庁）×個人プランは原則不可としてエスカレーション）

## フェーズ2: 出発資産別プレイブックの適用

ブロックBの回答に応じて以下を適用する。**資産が複数ある場合は合成する**（開始工程は最も上流の資産に合わせ、他の資産は「要件定義への入力」として登録する）。

### P1: 現行システムあり（リプレース）
- 追加質問: 現行の不満点（=リプレースの動機）/ 現行ドキュメントの信頼度 / 並行稼働の許容
- 初動: **要件定義より先に** legacy-modernizer で機能棚卸し（廃止候補の特定が最大の価値）
- 必須採用: legacy-modernizer / migration-specialist / business-process-analyst
- 罠と対策: 「現行通り」を要件と認めない（必ずAC化）。バグまで踏襲しない（実装挙動と意図仕様を区別）

### P2: 既存コードベースあり（エンハンス）
- 追加質問: 設計書とコードの乖離度（どちらが正か）/ 触ってはいけない領域 / リグレッションテスト資産の有無 / 元の開発会社との関係
- 初動: 資産評価（コード規約・データモデル・テスト資産の現状把握）→ `/context-history` で既存前提を外部化 → 影響分析してから要件定義
- 構成調整: **本パックの規約よりも既存コードの規約を正とする**（coder系への明示指示をCLAUDE.mdに書く）。data-model-specialistは既存ERDの復元から開始
- 罠と対策: 設計書が古い場合はlegacy-modernizerを部分投入（改修対象領域だけ仕様復元）。リグレッション範囲の基準を最初に決める（Q基準に追記）

### P3: モック・プロトタイプだけある
- 追加質問: モックの作者と位置づけ（合意済みデザインか叩き台か）/ モックにない画面・帳票の存在 / 実装技術とモックの乖離（モックのUIライブラリで本実装するのか）
- 初動: requirements-analyst がモックから機能一覧とACを**逆生成**し、曖昧さ検出を全開で実施
- 罠と対策: **モックは「画面の見た目の合意」であって要件ではない**。裏側が全部空白であることを顧客と共有する: 業務ルール・異常系・権限・非機能・バッチ・帳票・データモデル。空白リストを最初の顧客確認事項にする
- 構成調整: frontend-coder はモック踏襲を正とするが「モックにない状態（エラー/空/ローディング）」の設計起票を義務化

### P4: 類似システム・参考サービスがある（「〇〇みたいなの」）
- 追加質問: 参考サービスのどの部分を指しているか（全部ではないはず）/ 自業務と参考サービスの業務の違い / 参考サービスに触れる環境があるか
- 初動: business-process-analyst + requirements-analyst で参考サービスの機能棚卸し → Must/Want仕分け → **自業務との差分こそが本当の要件**として差分定義
- 罠と対策: 「あれと同じで」は要件ではない（同じにできない部分＝データ・規模・業務ルールで必ず崩れる）。デザイン・文言の模倣は著作権・利用規約の観点で「要法務確認」起票。競合の裏側仕様は推測になるため【推定】マーカー管理
- 見積り注意: 参考サービスの完成度を基準に期待値が膨らみがち → estimation-specialist の前提条件に「参考サービスとの差異一覧」を必須添付

### P5: RFP・要求仕様がある（標準ケース）
- 初動: requirements-analyst でRFPの曖昧さ検出（4分類）→ `/requirements-definition`
- 罠と対策: RFPの要求は「書いた人の解釈」を経ている。出典（誰の要求か）を遡って記録する

### P6: ビジネス目的だけある（超上流）
- 追加質問: 目的の定量化は可能か（KPI）/ 予算枠と投資判断者 / 「作らない」も選択肢か
- 初動: **要件定義に入らない**。business-process-analyst で業務・課題の構造化 → システム化する/しない/運用で解決の線引き → estimation-specialist で概算（±50%）→ 投資判断材料の提出まで
- 構成調整: 共通フレームの企画プロセス相当。工程スキルは `/requirements-definition` 以降を保留。契約は準委任を推奨（請負ならスコープ確定リスクをleader経由で人間へ）
- 罠と対策: いきなり画面の話を始めない（顧客はモックを見ると要件が決まった気になる）。PoC/プロトタイプ提案時は「捨てる前提か育てる前提か」を最初に合意

## フェーズ3: 構成の決定（決定表＋プレイブック差分）

共通決定表（規模・帳票・バッチ・連携・規制による採用/保留/除外は従来通り）に、フェーズ2のプレイブック指定を上書き適用し、一覧でユーザー承認を得る:

| 条件 | 構成judgment |
|---|---|
| 常時採用（削減禁止） | leader / documentation-specialist / data-model-specialist(DBあり) / code-reviewer / コンテキスト管理スキル / QCD基準 |
| 規模〜3人月 | requirements-analyst・test-engineer をメイン兼務、estimation/business-process をleader兼務に統合 |
| 帳票なし / バッチなし / 外部連携なし | report-specialist / batch-specialist / api-designer をそれぞれ除外・保留 |
| インフラが顧客側/定型 | infra-coder 除外 |
| 保守受託なし | incident-responder 保留（総合テスト開始時に復帰） |
| 性能要件が緩い | quality-performance 保留（Q-04/Q-08の担保先を code-reviewer に付け替え） |
| Excel納品指定 | /excel-deliverables 採用 |
| 業種規制あり | security-compliance に規制チェックリストを追記 |
| 機能ドメイン3つ以上＋並行開発 | **フィーチャーチーム運用**を提案（docs/standards/feature-team.md。ドメイン専任coderの複製、横断エージェントは複製禁止） |
| 大規模（20人月〜 / 多チーム / 協力会社あり / 監査対応あり） | **enterprise-controls を適用**（docs/standards/enterprise-controls.md。破壊防止3層・監査証跡・職務分掌・協力会社統制・ループ統制の適用チェックを実施） |
| ループ運用（定期実行・自動反復）の予定あり | enterprise-controls §B の開始条件チェックリストを適用（**ハーネス未整備でのループ化は禁止**） |
| タスク管理ツール利用あり | /pm-sync セットアップ＋正本ルールの確定（.mcp.json.template から接続） |
| 2案件目以降の組織展開 | ai-dev-standardizer 必須（初案件はleader兼務可）＋ /effort-compression のベースライン設定 |

- 除外は `.claude/agents.disabled/` への移動＋**復活条件**の記録（削除しない）

## フェーズ4: 適用

1. `CLAUDE.md.template` → `CLAUDE.md` 生成（出発資産と適用プレイブック名を明記。埋まらない項目は【要確認】起票）
2. 除外エージェントの移動・QCD基準の全IDテーラリング確定（ai-dev-standardizer主導）・確定値のCLAUDE.md転記
3. docs骨組み生成: `docs/open-questions.md` / `docs/issues.md` / `docs/decisions/` / `docs/context-history/`（初期LATEST.mdに本セットアップを記録）
4. プレイブック固有の初期成果物を登録（P1:棚卸し表の器 / P3:モック空白リスト / P4:差分定義表 / P6:線引き表）。**この時点で判明している曖昧さは全件、Q-ID＋影響範囲つきで open-questions に起票する**（曖昧なスタートの案件ほど、初期起票の量がそのまま仮定管理の土台になる。正本: docs/standards/assumption-management.md）
5. テーラリング記録（採用/保留/除外＋根拠＋復活条件）を CLAUDE.md へ（ai-dev-standardizer）
6. 開発規約の準備: 技術スタック確定済みなら `/dev-standards` を初動リストに入れる（**coder稼働前に規約正本を作る**。未確定なら /tech-stack-selection の完了を規約生成のトリガーとして記録）。テストランナー（vitest/jest等）の設定には `.claude/worktrees/**` の除外をデフォルトで含める（並行セッションのworktree成果物をテストに巻き込まない）
7. 工数圧縮の土台: `/effort-compression` のStep1-3（作業棚卸し・移管度マップ・ベースライン設定）をセットアップ期に実施（後からベースラインは作れない）
8. **AIセキュリティ統制の適用（省略不可）**: `docs/standards/ai-security-baseline.md` のプラン別チェックリストを実施 —
   共有 settings.json への deny 設定（.env・鍵・secrets の読み取り禁止等のスニペット適用）/ MCP許可リストの確定 /
   秘密情報の置き場所ヒアリングと deny への追加 / 報告免責ルールのCLAUDE.md明記。
   Enterprise なら managed settings への移設を管理者タスクとして起票、個人プランなら業務利用承認の記録を確認。
   **このステップが完了するまで coder 系エージェントを稼働させない**

## フェーズ5: 検収と最初の一手

- [ ] CLAUDE.md の {{ }} が全て埋まっている（【要確認】は起票済み）
- [ ] 出発資産と適用プレイブックが記録されている
- [ ] QCD基準に「仮基準」が残っていない
- [ ] 除外エージェント全件に復活条件がある
- [ ] AIセキュリティ統制（プラン別チェックリスト＋deny設定）が適用済み。個人プランの場合は業務利用の人間承認が記録されている
- [ ] 人間がテーラリング一覧を承認済み

最初の一手はプレイブックの「初動」をそのまま提示する（P1:現行棚卸し / P2:資産評価 / P3:モック逆生成＋空白リスト / P4:参考サービス棚卸し / P5:RFP曖昧さ検出 / P6:業務構造化。複数資産なら最上流から）。

## 原則

- 回答を推測で埋めない。「とりあえず全部入り」を提案しない
- 出発資産の見立てを間違えたと気づいたら、途中でもプレイブックを切り替えてテーラリング記録に残す
- このスキルの実行自体が最初のADR（`docs/decisions/YYYYMMDD-project-setup.md`）になる
