# 基本設計スキル

基本設計書一式を `docs/02-design/basic/` 配下に作成・更新する。**顧客がレビュー・承認する文書**なので、実装詳細ではなく「外から見える振る舞い」を書く。

## 前提条件（満たさない場合は着手しない）

- 対象機能の機能一覧・ACが確定または顧客確認中である
- 非機能要件に仮置き以上の値が入っている
- 未確定要件に依存する設計は、依存箇所に【要件未確定: Q-XXX】マーカーを付けること

## 手順

1. 対象機能（機能ID）を確認する
2. 方式レベルの設計（システム構成・連携方式）は architecture-guardian にレビューさせる
3. 機能単位で作成:
   - `docs/02-design/basic/screens/<画面ID>.md` — 画面設計書
   - `docs/02-design/basic/interfaces/<IF-ID>.md` — 外部IF・API設計書
   - `docs/02-design/basic/reports/<帳票ID>.md` — 帳票設計書（該当時）
4. テーブル定義・ERDは **data-model-specialist に委譲**（`/erd-update` 参照）
5. 認証・決済・個人情報を扱う機能は security-compliance のレビューを必須とする
6. **セキュリティ構成図（`docs/02-design/security-architecture.md`）を security-compliance に作成させる**（主管はsecurity-compliance。信頼境界・境界ごとの防御・**未適用の保護の明示**・攻撃面一覧・秘密情報の置き場。アーキテクチャ構成図は「何がどう繋がっているか」であり、「どこが信頼境界で、何で守られていて、何が守られていないか」はこの図でしか読めない。基本設計の完了条件 — UIありのHTMLモックゲートと同格）
7. **UIを持つ案件では、画面設計書と対になるHTMLモックを生成する**（後述「HTMLモック（UIあり案件のみ）」）。判断の担い手は `/project-init` の体制ギャップヒアリング（ブロックD-18）で決める: ui-ux-designer採用時はそちらが調査・複数案・提案を担う。保留時はfrontend-coder + requirements-analystがui-ux-designerの原則チェックリストを直接適用する
8. **テスト計画のシフトレフト（W字）**: 基本設計の完了前に `/test-planning` を呼び、結合テスト仕様書ドラフト（基本設計ベース）を作成する。テスト不能な設計（観測できないIF挙動・再現できない連携条件）は完了前に設計へ差し戻す
9. documentation-specialist に版数・変更履歴・顧客提出体裁を整えさせる（各設計書の冒頭に「レビュアー向けサマリ」欄 — deliverables-catalog の共通規約）

## 図の扱い（正本/ビュー基準）

設計書内の図は deliverables-catalog「図の正本/ビュー基準」に従う: シーケンス図・ER図・状態遷移図はMermaidのままでよい。**業務フロー図・DFD・機能関連図・構成図（アーキテクチャ/インフラ/セキュリティ）はHTML生成ビュー必須**（ノード10超 or エッジ交差が避けられない図も同様）。正本Mermaid/Markdownから一方向生成し、project-plan.html のデザイン言語（チップ・カード・モノクロSVG・絵文字禁止）を踏襲、正本と同ディレクトリに置く。

## 画面設計書テンプレート

```markdown
# <画面ID> <画面名>            版数: / 変更履歴:

対応機能ID: F-XXX / 対応AC: AC-XXX-01, ...

## 画面概要・利用者ロール
## 画面レイアウト（ワイヤー or 図の参照）
## 項目定義
| No | 項目名 | 種別(表示/入力/ボタン) | 型・桁 | 必須 | 初期値 | 活性制御 | バリデーション | 備考 |
## 画面遷移（遷移元/遷移先/条件）
## イベント定義（操作→処理→結果。エラー時挙動を必ず含む）
## 権限（ロール×操作のマトリックス）
## メッセージ一覧（メッセージID・文言・種別）
```

## HTMLモック（UIあり案件のみ）

UIを持つ案件では、画面設計書のレビューを「文章を読んで承認」ではなく「**モックを見て承認**」に置き換える。
人間は文章から画面を脳内レンダリングできず、UIの認識ズレは実装後に発覚するのが最も高くつく。モック生成は生成AIの得意領域であり、数分で作れるものを工程として要求しないのはもったいない。

- **発動条件**: UIを持つ案件のみ（バッチ/API専業では発動しない）。project-init または本スキル開始時に判定する
- **担い手**: `/project-init` の決定表でui-ux-designerが採用されている案件は、調査（既存画面の参照パターン）→複数案実装→提案（理由つき推奨）の流れをui-ux-designerが担う。保留の案件はfrontend-coderが素のモックを作り、ui-ux-designerの原則チェックリスト（デザイン原則チェックリスト参照）を自分で当てる
- **生成**: 画面設計書1枚につき `docs/02-design/basic/mockups/<画面ID>.html` を1枚生成する。静的HTMLで十分（画面遷移はリンクで表現）。デザインシステムがあれば適用し、なければ素のHTMLでレイアウト・項目・導線の確認に絞る。**絵文字をアイコン代わりに使わない**（チープに見える。アイコンが必要ならモノクロのインラインSVGかCSS図形で）
- **前例のない画面（ゼロイチ）の扱い**: 参照できる既存パターンがない画面は、AIの単独出力だけで最終品質に届くと期待しない。複数案は叩き台として提示し、人間（顧客・自社デザイナー）の判断を仰ぐ工程を明示的に挟む
- **ゲート**: モックの人間承認を基本設計完了判定の条件とする（deliverables-catalog 参照）。承認記録は画面設計書の承認欄に「モック確認済み」を明記する
- **正本の扱い**: 正本はあくまで画面設計書（Markdown）。モックはレビュー用ビューであり、承認後の仕様変更は「設計書を更新 → モックを再生成」の一方向のみ（モック直編集で二重正本を作らない）
- モック一覧は `tools/docs_viewer.py`（ローカルWebビューア）から閲覧できる

対応機能ID: / 方向(送/受): / プロトコル: / 頻度・タイミング:

## セキュリティ構成図テンプレート（`docs/02-design/security-architecture.md` — 主管: security-compliance）

```markdown
# セキュリティ構成図              版数: / 変更履歴: / 承認: （ロールID）

## レビュアー向けサマリ
（共通規約に同じ）

## 信頼境界図（Mermaid正本 → HTML/SVG承認ビジュアルを一方向生成）
（インターネット／エッジ（CDN・WAF）／アプリ層／データ層の入れ子と、境界ごとの防御。
 **未適用の保護（WAFなし・レート制限なし等）は点線＋チップで必ず描く** — 受容済みリスクと検討漏れの区別）

## 攻撃面一覧
| エンドポイント/入口 | 公開範囲 | 認証 | 入力検証 | 対応AC/テスト番号 | 備考（未保護の理由・受容記録） |

## 秘密情報の置き場と流れ
（環境変数・Secrets・リポジトリに置かないもの・本番との分離）

## 受容済みリスク（Q-09 判定の根拠。顧客合意の記録への参照つき）
```

## 完了条件

- 全項目定義に型・桁・必須が埋まっている（「TBD」を残さない。未確定は要件未確定マーカーで根拠を示す）
- 全画面・IFがACとトレース可能（対応AC欄が埋まっている）
- エラー時挙動が全画面・全IFに定義されている
- 顧客レビュー予定日と承認記録欄がある
- UIあり案件: 全画面のHTMLモックが存在し、人間が**モックを見て**承認した記録がある（設計書の文章レビューで代替しない）
- セキュリティ構成図（security-architecture.md）が存在し、security-compliance の承認と人間の確認記録がある（未適用の保護が明示されていること）
- 結合テスト仕様書ドラフト（基本設計ベース）が作成されている（W字対応・/test-planning）
